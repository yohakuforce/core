## 1. MUST — 公開ブロッカー (これ無しでは出せない / 出すべきでない)

### M1. lint を green にする  ★最優先・最大の不確実性
- **なぜ**: CI が `npm run lint` をゲート。現状 71 errors で **push した瞬間 CI が赤**。
- **内容**: エラーは新 Phase コード (src/diff, domains, explain-prompts, coverage 等) に集中。
  rule は style 中心 (`useTemplate` / `useNamingConvention` / `noUnusedTemplateLiteral` /
  `noNonNullAssertion`) + 一部 correctness (`noVoidTypeReturn`)。
- **手順**:
  1. `biome check src tests --fix` で safe fix を一括適用
  2. `--fix --unsafe` は差分を見て選別適用 (テンプレートリテラル等)
  3. 残り (命名規則 / 非 null 断言 / void return) を手で修正
  4. 修正後 `npm test` を回し回帰なしを確認
- **受け入れ基準**: `npm run lint` exit 0、`npm test` 497 passed 維持。
- **想定**: 1〜2h (auto-fix で大半、手修正が残る)

### M2. LICENSE / NOTICE を公開パッケージに含める
- **なぜ**: Apache-2.0 は NOTICE の同梱を要求。現状 tarball に無く、ライセンス不備のまま公開になる。
- **手順 (いずれか)**:
  - (a) `packages/core/LICENSE` `packages/core/NOTICE` を root からコピー (シンプル)
  - (b) `scripts/copy-assets.mjs` に root の LICENSE/NOTICE を dist 隣へコピーする処理を追加し `files` を調整
  - → (a) を推奨 (monorepo の各 publish パッケージが自前で持つのが npm 慣習)
- **受け入れ基準**: `npm pack --dry-run` の Tarball Contents に `LICENSE` と `NOTICE` が出る。

### M3. 再ビルドして Phase 2 配布物を dist に反映
- **なぜ**: 現 dist は Phase 2 前。`yohaku init --bootstrap` で配布される scaffold に
  新スラッシュ5件・local-llm/README.md が含まれない。
- **手順**: `npm run build` (scaffold→dist コピーを含む) を実行。
- **受け入れ基準**: `npm pack --dry-run` に下記が含まれる:
  - `dist/scaffold/.claude/commands/yohaku-{explain-prompts,html-write,serve,diff-html,coverage-import}.md.eta`
  - `dist/scaffold/prompts/local-llm/README.md`

### M4. 公開 tarball の実消費スモークテスト
- **なぜ**: 「ビルドが通る」と「インストールした人が動く」は別。アセット欠落は tarball を実際に
  入れて初めて分かる (今回の LICENSE 欠落もこれで気づける類)。
- **手順**:
  1. `npm pack` で `.tgz` を生成
  2. 一時ディレクトリで `npm i -g ./yohakuforce-core-0.5.0.tgz`
  3. 別の空 sfdx プロジェクトで `yohaku init --bootstrap` → `graph build` → `render --format md,html` → `serve`
  4. ホームが開き、新スラッシュ .eta が配置されることを確認
- **受け入れ基準**: クリーン環境でスモークテスト 5 手順が PASS。

### M5. branch → main マージ → push → CI green
- **なぜ**: 公開の前提。work は feature branch に隔離されたまま。
- **手順**: M1〜M4 を branch 上で確定 → main へマージ → push → GitHub Actions が green。
- **受け入れ基準**: main の CI が lint+test 共に green。coverage ゲート (80%) があれば新コードも充足。
- **注意**: coverage 閾値が CI で実際に効くか要確認 (効く場合、新 Phase コードの 80% 到達を確認)。

### M6. `npm publish` + GitHub Release  ★koya 承認・koya 実行
- **なぜ**: これが「公開」そのもの。koya-brain ルール上 **対外発信・課金判断は koya 必須**。
- **手順**: `npm version` 確認 (0.5.0) → `git tag v0.5.0` → `npm publish --access public` → GitHub Release 作成 (CHANGELOG 転記)。
- **受け入れ基準**: `npm view @yohakuforce/core version` = 0.5.0、GitHub に v0.5.0 tag/release。
- **AI の範囲**: M5 まで「publish 可能な状態」に仕上げる。publish コマンド実行は koya が承認・実行。
