# 提案: AI に「使うべきツールを使う」よう誘導する仕組み

**ステータス**: 提案 (別 Issue 化推奨)
**スコープ**: HTML 設計書パイプライン (Phase 0〜7) の対象外
**起点**: 2026-05 の yohaku/core レビュー (`Inbox/yohakuforce-coreレビュー.md`)

---

## 問題設定

knowledge graph が存在しても、AI (Claude Code / Codex 等) が反射的に `grep`/Read で
直接ソースを探索してしまい、せっかくの構造化資産が活用されない事象が観測された。

レビュー本文より:
> 今日の会話を振り返ると、`yohaku graph query` を一度も使わず、ひたすらファイルを直接
> Read・Grep していました。「メール上限エラーをデバッグしたい」という問いに対して、
> グラフを引くより `grep "sendEmail"` のほうが反射的に速かった。

整理すると、「問いの種類」によって最適手段が違う:

| 問いの種類 | 適したアプローチ |
|---|---|
| このオブジェクトを変えたら何が壊れる？ | グラフ (関係性の網羅) |
| このトリガはどこから呼ばれている？ | グラフ (探索) |
| なぜこのメソッドが 101 SOQL を出すのか？ | 生ソース (意味) |
| このバッチはどう動いているか？ | 生ソース (挙動) |

つまり、**「何を調べるか」をグラフで高速化、「どう動くか」をソースで確認** が役割分担。
ただし AI は問いの種類を自動判別せずに反射で動くため、入り口の設計が必要。

---

## 候補案 (3 つ、トレードオフ込み)

### 案 A: ヒューリスティック「使いたくなる入口」

- `/impact <name>` `/explain <name>` のスラッシュ系を充実させ、自然と最初の一手として
  使われる UX にする
- 強制せず、便利だから使われる方向

**長所**: 違和感なし、低リスク
**短所**: 反射で grep を選ぶ習慣は変わらない可能性

### 案 B: hook で `--edit` 前にグラフ参照を促す

- Claude Code の `PreToolUse` フックで「ファイル編集前に `yohaku impact <FQN>` を呼ぶか?」
  をユーザに問う / AI に提案する
- 編集対象の FQN を hook が graph で逆引きして必要そうなコマンドを提示

**長所**: 自然に AI が「影響範囲を確認した上で編集する」フローになる
**短所**: hook は Claude Code 依存。Codex/Antigravity では別実装が必要

### 案 C: CLAUDE.md に強制ルール

- 「影響範囲の質問には必ず `/impact` を呼ぶ」「複数ファイル編集前に `yohaku impact`」と
  明示的ルール化
- 違反は HUMAN_MANAGED で抑止できる前提だが、AI が自発的に守るかは別問題

**長所**: 文書一発で済む
**短所**: レビューの本文にあった通り「ルールが書いてあっても従うかはその場の判断次第」

## 推奨アプローチ (組合せ)

1. **案 A (Quick win)**: `/yohaku-html-build` / `/yohaku-domains` を Phase 6 で出した。
   HTML ホームから「コピー可能な `yohaku impact ...` コマンド」を見せることで、
   レビュー中に自然に手が動く動線を作る。
2. **案 B**: Mac mini 到着後、`PreToolUse` hook の prototype を作る。**最初は提案 only、
   強制しない**。
3. **案 C**: 上記 2 が機能してから、軽量ルールとして CLAUDE.md に追記する。順番は重要
   (ツールが揃わない状態でルール先行は守られない)。

---

## やらないこと (明示)

- **案 A/B/C を Phase 0〜7 のスコープに混ぜない**。HTML 設計書品質に集中する判断。
- 「絶対に grep を禁止する」ルールは作らない (バグ調査では grep が正しい)。
- ML / 強化学習による「AI の判断改善」は本プロジェクトの射程外。
