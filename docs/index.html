余白フォースyohakuforce suite
A three-layer foundation for AI-run delivery work

AIに任せられる仕事は、
余白フォースに任せる。

余白フォースは、受託・社内プロジェクトの「文脈の収集」「決定的な足場の生成」「進行の管理」を担う三層構造です。ブレない足場の上でAIが取りこぼしなく実務を回し、人は本質的な思考と意思決定、そして余白からの発想に集中する。

コア — 決定的な知識足場（OSS） コンテクストハブ — 文脈の収集・提供（OSS） AIマネージャー — 進行管理の5能力（社内）

core

コア / @yohakuforce/core

Salesforce のメタデータを SQLite の知識グラフに変換し、そこから決定的に Markdown ドキュメントを生成する CLI。LLM を呼ばないのが核心 — 同じ入力からは常に同じ出力になり、出所（provenance）を Tracked<T> で管理する。

npm: @yohakuforce/corev0.5.0Apache-2.0CLI: yohaku

01何ができるか

graph

知識グラフ化

force-app/ の SObject・項目・入力規則・Flow・Apex・依存関係を SQLite グラフに格納。SQL で横断クエリ可能（差分ビルド対応）。

render

決定的ドキュメント生成

システム全体像・SObject 個別・Flow・Apex を Markdown 化。同一入力→同一出力をゴールデンテストで保証。

diff

変更差分の抽出

Git ref 間でメタデータ変更を抽出し、change-summary の材料にする。

onboard

オンボーディング支援

ロール別の進行記録・FAQ 抽出で、プロジェクト立ち上げの文脈を蓄積。

validate

グラフ検証

JSON グラフのスキーマ整合を検証。CI に組み込み可能。

context

文脈注入（opt-in）

コンテクストハブから抽象化済み文脈を取得し、explain / change-summary の生成材料にする。

  1. インストール

    npm から導入。CLI は yohaku。

  2. 初期化 & グラフ構築

  4. （任意）ハブ文脈を有効化

05設計原則

3層分離
決定的処理はコアで担い、AI 推測は混ぜない。
決定的 I/O
同一入力 → 同一出力をハッシュ（ゴールデンテスト）で検証。
force-app は読むだけ
正本を書き換えない。グラフは CLI 経由でしか更新しない。
文脈注入の設計。コアは LLM を呼ばない決定的足場。注入は「ContextProvider が決定的に取得 → 外部AI(explain-writer 等)が材料化」の2層構造で、顧客固有名・PII・シークレットは抽象化され生データが成果物に漏れない。

ai-manager

AIマネージャー / AI-Project-Manager

コンテクストハブの文脈をもとに、プロジェクト進行を5つの能力で回す。会議や課題からタスクを起こし、担当に割り当て、日報で追跡し、遅延を検知し、経営サマリにまとめる。文脈はハブから REST で取得し、進行データは自前の PostgreSQL に保持する。

AI-Project-ManagerprivateREST :8001PostgreSQL

015つの能力（詳細）

Plan · 計画extract / import
会議メモ（ハブの get_meeting の抽出タスク）と Backlog/Redmine 課題（ステータス絞り込み可）からタスクを自動生成し、プロジェクトに取り込む。初期開発案件でも「最初からAIがタスクを切る」状態をつくる。
Assign · 割当draft / confirm
メンバーのスキル・稼働状況・負荷をふまえて担当案（draft）を生成。PM が承認 / 却下して確定する。
Track · 進捗追跡report / analyze
日報テンプレを生成・配信し、回答を集約。回答を分析してブロッカーを抽出する。
Alert · アラートscan
プロジェクトをスキャンし タスク遅延 / メンバー過負荷 / 顧客（日報）未回答 を検知して通知。確認・解決の状態管理つき。
Overview · 経営サマリsummary / phase
タスク状態・日報提出状況・アクティブアラートを集約した日次サマリと、フェーズ進捗（計画対実績）を生成。

03いつ・何が動くか — 一日の運用

内蔵スケジューラ（APScheduler）が設定時刻に各能力を自動実行する。時刻は設定 GUI（/settings）で自由に調整でき、既定は 経営サマリ 7:00 / 日報配信 8:00 / アラートスキャン 30 分間隔。処理の順序は時刻設定に関わらず常に固定（経営サマリ→日報生成→日報配信→アラート）で、ユーザーがどんな時刻を設定しても入れ替わらない。

07:00毎朝

経営サマリ生成Overview · 自動

前日のタスク状態・日報提出状況・アクティブアラートを集約し、AI が日次サマリとフェーズ進捗（計画対実績）を生成。PM はこれを見て一日を始める。

POST /overview/daily-summary · /overview/phase-progress
08:00毎朝

日報テンプレ生成 → 配信Track · 自動

その日の担当タスクから質問を動的に組み立て、メンバーごとの日報を生成。各メンバーのチャネル（Slack DM / Sheets / ローカル）へ配信する。

POST /track/generate-templates → /track/deliver
日中随時

回答の収集メンバー

メンバーが配信された日報に回答。AI マネージャーが受信して記録し、提出済みに更新する。

POST /track/submit-responses
回答後自動

AI 分析 → ブロッカー抽出Track · 自動

回答を AI が解析し、進捗サマリとブロッカーを抽出。検出したブロッカーは以降のアラート・経営サマリに反映される。

POST /track/analyze
30分毎終日

アラートスキャンAlert · 自動

タスク遅延・メンバー過負荷（推定残時間 40h 超）・日報未回答を検知し、アラートを生成して通知チャンネルへ。確認・解決の状態管理つき。

POST /alerts/scan
自動実行（設定時刻 / 間隔で起動） 人が関与（回答・承認・確認）
図 5 — 一日の運用ループ。設定した時刻に各能力の REST エンドポイントが実行される。
Plan・Assign はイベント駆動。計画（Plan）と割当（Assign）は時刻ではなく新しい会議メモ・課題が増えたとき（プロジェクト発足時・スプリント開始時など）に実行する。Assign だけは AI が割当案を作り、PM が承認 / 却下して確定する人の関門を挟む。

04日報のライフサイクル — 作成・共有・収集・分析

① 作成
タスクから動的生成
② 配信（共有）
メンバーのチャネルへ
③ 収集
回答を記録
④ AI 分析
サマリ + ブロッカー

図 6 — 日報は「タスク状態から作られ、各メンバーへ配信され、回答が集約され、AI が分析する」一連で回る。
① 作成dynamic template
メンバーの担当タスクごとに「進捗率（0〜100）」と「ブロッカー有無」の質問を生成。ブロック中のタスクには「顧客確認の要否」を追加。担当タスクが無ければ自由記述のみ。当日分が既にあるメンバーは二重生成せずスキップ。
② 配信（共有）multi-channel
メンバーの external_id を Slack のチャネル / DM として解決して配信。未設定なら既定チャネルへフォールバック。配信は Slack / Google Sheets / ローカルファイルから選択。
③ 収集submit
メンバーの回答を受信して日報に記録、提出済みへ更新。配信済みで未回答のものは Alert の「日報未回答」スキャンが拾う。
④ AI 分析analyze
回答を AI が解析して進捗サマリを生成し、ブロッカーを構造化抽出。「なし」以外のブロッカー回答は経営サマリ・アラートの材料になる。

AI が回し、人は要所だけ。取込・タスク化・日報配信・分析・アラート・サマリは自動で回る。人が手を入れるのは「割当の承認」「日報への回答」「アラートの解決判断」だけ — 進行管理の定型は AI に任せ、人は判断に集中する。
順序はユーザー操作で壊れない。調整できるのは時刻だけで、実行順序は内部で固定。経営サマリと日報配信を同じ時刻にしても 1 ジョブにまとめて正しい順に逐次実行し、スキャン間隔を極端に短くしても多重起動・滞留しないよう同時実行を 1 つに集約する。不正な時刻は既定値へ丸めて停止しない。

02ユーザー目線の一連の流れ

  1. プロジェクト発足・文脈集約

    Slack・Backlog・会議メモ等をハブに取り込む。Slack は既存スクレイピング結果を push、Backlog/Redmine は API、会議メモは登録時に自動でタスク抽出。

  2. 初期タスクの自動生成

    AIマネージャーが会議・課題からタスクを起こす。初期開発案件でも最初からタスクが切られた状態に。

  3. 割当と進行管理

    担当案の生成・承認、日報の配信と分析、遅延・過負荷の検知。日々の進行が回る。

  4. 成果物の生成（コア）

    Salesforce 案件ではコアが知識グラフから設計ドキュメントを決定的に生成。ハブの文脈を opt-in で材料化できる（生データは転記しない）。

  5. 経営サマリ

    タスク・日報・アラートを集約した日次サマリで全体像を把握。人は意思決定と余白からの発想に集中する。
