01何ができるか

graph

知識グラフ化

force-app/ の SObject・項目・入力規則・Flow・Apex・依存関係を SQLite グラフに格納。SQL で横断クエリ可能（差分ビルド対応）。

render

決定的ドキュメント生成

システム全体像・SObject 個別・Flow・Apex を Markdown 化。同一入力→同一出力をゴールデンテストで保証。

diff

変更差分の抽出

Git ref 間でメタデータ変更を抽出し、change-summary の材料にする。

onboard

オンボーディング支援

ロール別の進行記録・FAQ 抽出で、プロジェクト立ち上げの文脈を蓄積。

validate

グラフ検証

JSON グラフのスキーマ整合を検証。CI に組み込み可能。

context

文脈注入（opt-in）

コンテクストハブから抽象化済み文脈を取得し、explain / change-summary の生成材料にする。

ai-manager

AIマネージャー / AI-Project-Manager

コンテクストハブの文脈をもとに、プロジェクト進行を7つの能力で回す。会議や課題からタスクを起こし、担当に割り当て、朝会で前日を振り返り、日報で追跡し、未提出を催促し、遅延を検知し、当日を総括する。AI が定型を回し、判断が必要な要所はリーダー確認ゲートで人が決める。文脈はハブから REST で取得し、進行データは自前の PostgreSQL に保持する。

02いつ・何が動くか — 一日の運用タイムライン

内蔵スケジューラ（APScheduler）が設定時刻に各フェーズを自動実行する。時刻（時・分）は設定 GUI（/settings）で自由に調整でき、既定は 朝会 9:00 / 日報 14:00 / 催促 17:00 / 総括 17:30 / アラート 30 分間隔。処理の順序は時刻設定に関わらず常に固定（standup → report_generate → report_deliver → report_reminder → wrap_up → alert_scan）で、どんな時刻を設定しても入れ替わらない。リーダー確認後の全体分析（final_analysis）だけは時刻ではなく人の確認で発火する。

図 4 — 一日の運用ループ。9:00 朝会から 17:30 総括までが自動で進み、リーダーがゲートを解決すると全体分析が発火する。
Plan・Assign はイベント駆動。計画（Plan）と割当（Assign）は時刻ではなく新しい会議メモ・課題が増えたとき（プロジェクト発足時・スプリント開始時など）に実行する。日々のタイムラインとは独立。
順序はユーザー操作で壊れない。調整できるのは各フェーズの時刻（時・分）だけで、実行順序は内部で固定。複数フェーズを同じ時刻にしても 1 ジョブにまとめて正しい順に逐次実行し、スキャン間隔を極端に短くしても多重起動・滞留しないよう同時実行を 1 つに集約する。不正な時刻 / 分は既定値へ丸めて停止しない。1 プロジェクト / 1 ステップの失敗は記録のみで他に波及しない。

03リーダー確認ゲート — 人間判断のトリガー

時刻では進められない「人間の確認・判断」を待つ仕組み。総括（17:30）でゲートが起票され、リーダーが解決した瞬間に後続処理が発火する。確認が翌日になっても保持される（PostgreSQL 永続化）。

宛先。リーダー向けの共有・打診は設定の通知チャネル（例 Slack #ai-pm-alerts）へ、本人向けの催促は各メンバーの DM へ送る。

② 配信（共有）multi-channel
メンバーの external_id を Slack のチャネル / DM として解決して配信。未設定なら既定チャネルへフォールバック。Slack / Google Sheets / ローカルから選択。
③ 収集submit
回答を受信して記録、submitted へ更新。配信済みで未回答のものは 17:00 催促と「日報未回答」アラートが拾う。
④ AI 分析analyze
回答を解析して進捗サマリを生成し、ブロッカーを構造化抽出。「なし」以外のブロッカーは総括・全体分析・アラートの材料になる。

02ユーザー目線の一連の流れ

  1. プロジェクト発足・文脈集約

    Slack・Backlog・会議メモ等をハブに取り込む。Slack は既存スクレイピング結果を push、Backlog/Redmine は API、会議メモは登録時に自動でタスク抽出。

  2. 初期タスクの自動生成

    AIマネージャーが会議・課題からタスクを起こす。初期開発案件でも最初からタスクが切られた状態に。

  3. 割当と進行管理

    担当案の生成・承認、日報の配信と分析、遅延・過負荷の検知。日々の進行が回る。

  4. 成果物の生成（コア）

    Salesforce 案件ではコアが知識グラフから設計ドキュメントを決定的に生成。ハブの文脈を opt-in で材料化できる（生データは転記しない）。

  5. 経営サマリ

    タスク・日報・アラートを集約した日次サマリで全体像を把握。人は意思決定と余白からの発想に集中する。
