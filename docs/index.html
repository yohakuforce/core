core

コア / @yohakuforce/core

Salesforce のメタデータ（force-app/）を SQLite の知識グラフに変換し、そこから決定的にドキュメントを生成する CLI。出力は読みやすい Markdown と、検索・図解つきの HTML ナレッジハブの 2 系統。コアは LLM を一切呼ばない — 同じ入力からは常に同じ出力になり（ゴールデンテストで保証）、出所（provenance）は Tracked<T> で追跡する。このタブだけで、導入から日常運用・AI 連携・設計思想まで分かるように書いている。

01何ができるか

graph

知識グラフ化

SObject・項目・入力規則・レコードタイプ・Flow・Apex・トリガ・LWC・権限・レイアウト・依存関係を SQLite グラフに格納。SQL で横断クエリ可能（差分ビルド対応）。

render md

決定的 Markdown 生成

システム全体像・SObject・Flow・Apex・トリガ・権限・入力規則を Markdown 化。同一入力→同一出力をゴールデンテストで保証。

render html

HTML ナレッジハブ

検索（⌘K）・依存図・業務フロー・ホットスポット・コンポーネント詳細を備えた静的サイトを生成。項目名は日本語ラベル主・API 名副で初見の利用者にも読める。自己完結（外部依存ほぼ無し）。

serve

ローカルプレビュー

生成 HTML を配信する軽量サーバ。--watch でソース変更時に再生成＋ブラウザ自動リロード（SSE）。

diff

リリースレビュー

Git ref 間のメタデータ変更を抽出し、変更点から各コンポーネント詳細へリンクした HTML レビューを出力。静的解析（SARIF）も取り込み可能。

domains

ドメイン分類

domains.yaml で業務ドメインを定義。グラフからヒューリスティックに初期生成し、未分類・多重所属を lint で検出。

coverage

テストカバレッジ

Salesforce のテスト結果（JSON）を取り込み、Apex/トリガの被覆率を HTML に表示。低カバレッジ順サマリも。

AI 連携

安全な AI 追記（opt-in）

空の AI ブロック向けにプロンプトを出力 → 外部 AI が埋めた JSON を html-write で安全反映。コア本体は決定的なまま。

03前提条件

Node.js ≥ 20
CLI 実行環境。グラフ生成は better-sqlite3 を使用（プリビルドバイナリ）。
Salesforce DX（任意）
--source org で組織から直接 retrieve する場合のみ sf CLI と認証が必要。ローカルの force-app/ を読むだけなら不要。
git（任意）
変更履歴セクションと diff に使用。無くてもプレースホルダにフォールバックする。

対応プラットフォーム（macOS / Linux / Windows）

純 Node 実装のため 3 OS で動作する（パスは node:path、git / sf は引数配列で起動し shell 非依存、serve は \ と / の両方を処理）。Windows でもコア機能は一通り動くが、次の前提に注意。

項目Windows での前提・挙動
better-sqlite3（ネイティブ）知識グラフ用。Node LTS（20 / 22）・x64 ならプリビルドバイナリが入る。合致しない構成では Visual Studio Build Tools + Python が必要になる（最大の前提）。
git / sf CLI変更履歴・diff（git）と --source org（sf）使用時のみ PATH に必要。無ければ該当機能だけ無効化され、致命的にはならない。
--source dx-mcp外部 MCP サーバを子プロセス起動するこの経路だけ、Windows の .cmd 解決に注意（実験的経路）。local / org 経路は影響なし。
生成物（md / html）静的ファイル。Windows でもそのまま開ける・yohaku serve で配信できる。
推奨環境。素の Windows + Node LTS で動作する想定。native ビルドまで含めて確実にしたい場合は WSL も使える（必須ではない）。各ハーネス（Claude Code / Codex / Antigravity / ローカル LLM）も Windows 版があり、yohaku を呼ぶだけなのでコア同様に動作する。

11AI で「意味づけ」を足す（任意・安全）

コアが生成する HTML には、決定的に書けない箇所（業務的意味づけ等）が空の AI 管理ブロックとして残る。外部 AI で埋める流れは次の 2 ステップ。コア本体は決定的なまま保たれる。

  1. プロンプトを書き出す

    yohaku explain-prompts --kind business-meaning,concerns \
  --output prompts.json

  2. AI の回答（fill.json）を安全反映

    マーカー付きブロックにだけ書き込み、人間の編集や他セクションは壊さない。

    yohaku html-write --input fill.json --dry-run   # 差分確認
yohaku html-write --input fill.json

12各 AI ハーネスでの使い方

yohaku は Claude Code / OpenAI Codex / Google Antigravity / ローカル LLM のいずれからでも使える。共通の前提は「CLI が唯一のソース」— どのツールでも yohaku render --format md,html 等が決定的に同じ出力を出す。スラッシュコマンドを持つのは Claude Code だけで、他は AGENTS.md やチートシート経由で CLI を直接呼ぶ。yohaku init --bootstrap が各ツール向けの導線（.claude/ / AGENTS.md / prompts/）を配置する。

ネイティブ統合。スラッシュコマンド＋専門サブエージェントが .claude/ に配置され、会話的に駆動できる（裏で CLI と知識グラフを呼ぶ）。配置コマンドは --profile で変わる。各スキルは argument-hint に沿って引数を取る（<…> 必須 / […] 任意 / a|b 選択肢、Claude Code の $ARGUMENTS として渡る）。

07権限・監査・プライバシー

認証X-Api-Key
全エンドポイント（health / version 除く）が X-Api-Key ヘッダ必須。キーは消費者（consumer）に紐づき、スコープ集合を持つ。未指定は 401（UNAUTHORIZED）。
スコープread / write / admin
read（検索・参照）/ write（取込・同期）/ admin（全権）。要求スコープを満たさなければ 403。admin は全スコープを内包。起動時の dev キー（環境変数）に一致するキーは DB 照会なしで ADMIN 相当になる（開発用）。
データ境界local-only
取り込んだ全テキストはローカル（SQLite ファイル / 自前 Postgres）に保存。MCP は stdio（同一マシン内通信）で完結し、生の顧客データがマシンから出ない。API キーは索引に保存されず、認証のみに使う（.env で差し替え→再起動でローテーション）。
Gmail トークンrefresh token
GMAIL_TOKEN_FILE に refresh token が入る。インボックスを読める鍵なのでパスワード扱い（コミット禁止）。失効は myaccount.google.com/permissions から。
誤取込の対処incident
①サーバ停止 → ②GET /projects/{id}/context 等で対象を特定 → ③SQLite から削除 or rm -rf data/ で全リセット → ④Gmail 由来ならラベルを外して full_resync でカーソルを追従。

#   + slack: status=completed items=12
#   + backlog: status=completed items=3
# Ingest-all complete. project_id='…' succeeded=2 failed=0
定期実行は2方式。①serve 常駐スケジューラ＝context-hub serve は起動時に各プロジェクトの有効な全ソースを syncInterval（最小5分）ごとに自動同期するジョブを登録する（inbox 監視も同様）。既定で ON、CH_SOURCE_SYNC_ENABLED=false で停止。②外部スケジューラ＝ingest all を launchd（macOS）/ cron / systemd / Windows Task Scheduler で定期キック（同梱 examples/launchd/ に plist・各OS例）。サーバ常駐なら①、再起動を跨いで確実に回すなら②。どちらか一方を選ぶ（両方だと二重同期）。1プロジェクト=1インスタンスで複数案件も混ざらない。

Windows 対応

Windows でも動作する。注意点は1つだけ：python.org 公式の Windows ビルドは sqlite3 が拡張ロード非対応のため、SQLite プロファイルでは sqlite-vec（ベクトル検索）が読み込めない。その場合 Context-Hub は自動で FTS-only（キーワード検索）モードに縮退し、取り込み・キーワード検索・MCP・REST は通常どおり動く（意味検索のみ無効、起動時に警告ログ）。

Windows でフル意味検索が要るとき。①conda / miniforge の Python（拡張ロード対応）を使えば SQLite プロファイルでもベクトル検索が有効、または②production プロファイル（PostgreSQL + pgvector）を使う（sqlite-vec 不使用なのでどの Python でも完全動作）。定期実行は Windows Task Scheduler で ingest all を回すか、serve 常駐スケジューラを使う。

02ユーザー目線の一連の流れ

  1. プロジェクト発足・文脈集約

    Slack・Backlog・会議メモ等をハブに取り込む。Slack は既存スクレイピング結果を push、Backlog/Redmine は API、会議メモは登録時に自動でタスク抽出。

  2. 初期タスクの自動生成

    AIマネージャーが会議・課題からタスクを起こす。初期開発案件でも最初からタスクが切られた状態に。

  3. 割当と進行管理

    担当案の生成・承認、日報の配信と分析、遅延・過負荷の検知。日々の進行が回る。

  4. 成果物の生成（コア）

    Salesforce 案件ではコアが知識グラフから設計ドキュメントを決定的に生成。ハブの文脈を opt-in で材料化できる（生データは転記しない）。

  5. 経営サマリ

    タスク・日報・アラートを集約した日次サマリで全体像を把握。人は意思決定と余白からの発想に集中する。
