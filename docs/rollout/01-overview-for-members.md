## 1. どんな問題を解こうとしているか

Salesforce プロジェクトの現場では、繰り返し同じ手間が発生します。

- **メタデータの全体像を把握できない**。ApexClass / Flow / 権限まわりが膨大になり、「誰が何を知っているか」に依存した属人化が進む。
- **新規参画者のキャッチアップに時間がかかる**。詳しい人に何度も聞かなければ動けない。
- **ドキュメントが陳腐化する**。手動で更新するコストが高く、実装との乖離が生じる。
- **リリース前の手動作業が網羅されているか不安**。何かを見落としていても気づきにくい。

これらは特定プロジェクトの問題ではなく、Salesforce 開発全体に共通する構造的な課題です。

## 2. yohakuforce とは何か

**Salesforce に特化した AI 駆動の OSS 基盤**です。

CLI ツール（`yohaku`）と Claude Code の連携により、上記の手間を自動化します。設計の核となる考え方は **3 層分離** です。

| 層 | 何をするか | 特徴 |
|---|---|---|
| 決定的処理層（`yohaku` CLI） | XML パース、グラフ構築、差分検出、テンプレート描画 | 同じ入力には必ず同じ出力。再現性が命 |
| AI エージェント層（Claude Code） | 意味分類、自然語サマリ、影響推測、レビュー観点の提示 | 判断と曖昧さを担う。生の XML データは読ませない |
| 人手補完層（開発メンバー） | 業務意図の供給、固有ルールの判断、最終承認 | AI が侵さない領域 |

**「同じ入力には同じ出力」という再現性が、この基盤の核です。** AI が揺らいでも、CLI が決定的に処理した結果をベースにするため、プロジェクト間で同じ作法が通用します。

## 3. 具体的に何ができるか

### オンボーディング支援（`/onboard`）

Claude Code 上で `/onboard` を実行すると、プロジェクトの全体像（オブジェクト構成・自動化・権限）を対話形式で説明します。新規参画者が「詳しい人に聞く」ではなく「AI に聞く」で済む状態を目指します。

### ドキュメント自動生成（`yohaku render` / `yohaku sync`）

`force-app/` 配下のメタデータから知識グラフ（SQLite）を構築し、Markdown ドキュメントを自動生成します。手動更新が不要になります。21 種類のメタデータに対応（Apex / Flow / Trigger / 権限 / Layout / LWC / Aura 等）。

### 差分の意味づけ（`yohaku diff`）

Git の差分を「データモデル変更」「自動化変更」「権限変更」「UI 変更」「ロジック変更」などの観点で自動分類します。レビュー時に「この変更は何に影響するか」がすぐにわかります。

### 手動作業の検出（`yohaku onboard / manual-steps`）

リリース前に必要な手動作業（本番データ移行・権限設定・参照整合性確認など）をルールベースで検出し、見落としを防ぎます。

### 設計書の AI 補完（`yohaku explain` + Claude）

既存の設計書に AI_MANAGED ブロックを設け、Claude が自然語サマリや業務観点の説明を書き込みます。人が追記した内容（HUMAN_MANAGED ブロック）は再生成時でも保護されます。

## 4. 何が変わるか

| 今まで | yohakuforce 導入後 |
|---|---|
| 新人がキャッチアップするたびに詳しい人が時間を取られる | `/onboard` を実行すれば基本的な全体像が把握できる |
| ドキュメントは手動更新・すぐ陳腐化 | `yohaku sync` 1 コマンドで自動更新 |
| リリース前のチェックリストは記憶と経験に依存 | 手動作業をルールベースで自動検出 |
| プロジェクトが変わると知見を持ち運べない | 同じ CLI・同じ作法を別プロジェクトにそのまま展開できる |
